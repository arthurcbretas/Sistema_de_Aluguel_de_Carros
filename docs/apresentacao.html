O que é o DriveLux?

💬 "O DriveLux é um sistema web completo de aluguel de carros. O cliente entra pelo navegador, escolhe um carro, pede o aluguel e, se quiser, solicita financiamento. A empresa analisa e aprova. Tudo online, sem precisar instalar nada."

Um sistema web onde o cliente acessa pelo navegador, escolhe um carro, faz o pedido de aluguel e acompanha tudo online.

Fazer Pedidos

Escolher carro, datas e opcionalmente pedir financiamento

Avaliar e Aprovar

A empresa analisa e gera contratos automaticamente

Financiamento

O banco associa crédito a quem solicitou

Cálculos Automáticos

Valor do aluguel = dias × diária do carro

Quem usa o sistema?

💬 "O sistema tem três tipos de usuário, cada um com sua própria tela e acesso diferente. O cliente aluga, a empresa avalia e aprova, e o banco cuida do financiamento. Ao fazer login, o sistema já redireciona cada um para o painel certo."

AtorO que fazTela
Cliente Cadastra-se, navega no catálogo de carros, cria pedidos, pode pedir crédito Dashboard do Cliente
Empresa Avalia pedidos (aprova ou rejeita), gerencia a frota de carros Painel Administrativo
Banco Vê contratos que pediram crédito, define valor, parcelas e juros Painel Bancário
Sistema Calcula valor do aluguel e gera contratos automaticamente —
Cada ator tem sua própria tela. Ao fazer login, o sistema identifica o tipo de usuário e redireciona para o painel correto.

Como funciona?

💬 "O fluxo começa no cliente: ele cria um pedido escolhendo carro e datas. A empresa recebe, analisa e aprova — gerando um contrato automaticamente. Se o cliente pediu financiamento, o banco entra também para definir as condições de crédito."

Regras de Negócio

💬 "Algumas regras importantes que o sistema garante automaticamente: o valor é calculado sem o usuário precisar digitar nada, o contrato é gerado sozinho ao aprovar, e cada perfil só acessa o que é seu."

  • O valor do aluguel é calculado automaticamente: dias × preço da diária
  • Pedido aprovado → contrato gerado automaticamente
  • O cliente pode marcar que necessita de crédito ao criar o pedido — essa decisão é definitiva
  • O banco só vê contratos onde o cliente pediu financiamento
  • Cada cliente pode ter no máximo 3 rendimentos e 3 empregadores
  • Login e CPF são únicos
  • Um automóvel pode pertencer a um Cliente, Empresa ou Banco
Ciclo do pedido: PENDENTE → APROVADO (gera contrato) ou REJEITADO. O cliente pode CANCELAR antes da avaliação.

Por que o Micronaut?

💬 "O Micronaut é uma alternativa moderna ao Spring Boot. A principal vantagem é que ele inicializa muito mais rápido e usa menos memória — o que facilita bastante durante o desenvolvimento. E como as anotações são parecidas com o Spring, a curva de aprendizado não é tão grande."

O Micronaut é um framework Java moderno, alternativa ao Spring Boot, focado em performance e leveza.

Rápido para iniciar

Inicia em ~2 segundos (Spring Boot leva ~5-8s)

Usa pouca memória

~70-100 MB contra ~150-300 MB do Spring

Detecta erros antes

Processa na compilação — erros aparecem antes de rodar

Parecido com Spring

Anotações similares — fácil para quem já conhece

O Micronaut foi escolhido pelo requisito do LAB de utilizar um framework diferente do Spring Boot, e se mostrou uma alternativa viável e produtiva.

Arquitetura do Sistema

💬 "O sistema segue a arquitetura MVC em camadas. O navegador fala com os Controllers, que chamam os Services onde ficam as regras de negócio, que por sua vez acessam o banco pelo Repository. Cada camada tem sua responsabilidade bem definida."

O sistema segue o padrão MVC em camadas — cada camada tem uma responsabilidade:

Modelo de Dados

💬 "Para representar os diferentes tipos de usuário, usei herança de classes com a estratégia JOINED no JPA. Isso significa que cada tipo — Cliente, Empresa e Banco — tem sua própria tabela no banco, mas compartilham os campos comuns de Usuário."

Segurança

💬 "A segurança é feita com JWT — o usuário faz login, recebe um token que identifica quem ele é e qual papel tem, e manda esse token em toda requisição. As senhas são armazenadas com BCrypt, ou seja, criptografadas — nunca em texto puro no banco."

O sistema usa JWT (JSON Web Token) para autenticação. O usuário envia login e senha, recebe um token com seu papel, e envia esse token em todas as próximas requisições.

3 papéis no sistema

PapelQuem éO que acessa
CLIENTEPessoa físicaCatálogo, pedidos, perfil
AGENTEEmpresa (DriveLux)Avaliar pedidos, gerenciar carros
BANCOInstituição financeiraContratos pendentes de crédito
A senha do cliente é armazenada com criptografia (BCrypt) — nunca salva em texto puro.

Desafios de Arquitetura

💬 "O maior desafio técnico foi a serialização de entidades JPA para JSON — quando o Hibernate tenta carregar relações Lazy fora da transação, dava erro. Precisei controlar exatamente quais campos são retornados em cada endpoint."

Serialização de entidades JPA para JSON

Entidades com relacionamentos Lazy carregados pelo Hibernate causavam erro ao serem convertidas para JSON fora da transação. Tive que usar anotações para controlar quais campos são serializados e, em alguns casos, retornar objetos intermediários em vez da entidade direta.

Comunidade e documentação do Micronaut

Por ser um framework mais novo, o Micronaut tem menos conteúdo disponível online. Precisei recorrer à documentação oficial e ao código-fonte do próprio framework para resolver problemas específicos.

Compatibilidade H2 com herança JOINED

A estratégia de herança JOINED no JPA exigiu fixar a versão do H2 (2.2.224) por incompatibilidades com o Hibernate 6. Sem isso, o schema não era gerado corretamente em desenvolvimento.

Mais Desafios Técnicos

💬 "Outro desafio foi o encoding de caracteres no Windows — o compilador Java assumia Cp1252 e corrompía acentos no build. E o sistema de autenticação do Micronaut é reativo, o que é diferente do Spring Security e exigiu adaptação."

Gerenciamento de ambientes (dev vs produção)

O Micronaut usa variáveis de ambiente para alternar perfis. No Windows, a variável se perdia entre sessões do terminal, fazendo o servidor tentar conectar no PostgreSQL de produção em vez do H2 de desenvolvimento.

Encoding UTF-8 no Windows com Gradle

O compilador Java no Windows assume Cp1252 por padrão. Strings com acentos no código-fonte eram corrompidas no build. Exigiu configuração explícita de encoding no Gradle.

Autenticação reativa no Micronaut Security

O AuthenticationProvider do Micronaut retorna um Publisher reativo (Project Reactor), diferente do modelo síncrono do Spring Security. Exigiu adaptação para trabalhar com Mono no fluxo de login.

Micronaut vs Spring Boot

💬 "Comparando os dois: Micronaut ganha em performance — inicia em 2 segundos e usa menos memória. Spring Boot ganha em comunidade e facilidade de aprender. Para projetos que precisam de performance, o Micronaut é uma excelente alternativa."

O que aprendi

💬 "As principais lições focado no Micronaut foram: a grande diferença no tempo de resposta para desenvolvimento se comparado ao Spring; como a validação no tempo de compilação salva tempo evitando erros em execução; e a simplicidade de gerenciar a aplicação pelas configurações YAML sem precisar de muitas classes extras."

  • O Micronaut é altamente produtivo — o tempo de inicialização ultrarrápido agiliza demasiadamente o ciclo de compilar e testar.
  • A configuração declarativa via arquivos YAML (application.yml) centraliza e reduz muito código boilerplate.
  • O framework valida queries de banco e injeção de dependências em tempo de compilação (AOT), antecipando erros.
  • Entender parte do modelo reativo e não-bloqueante de segurança do Micronaut Security trouxe uma nova perspectiva em relação a sistemas web.
